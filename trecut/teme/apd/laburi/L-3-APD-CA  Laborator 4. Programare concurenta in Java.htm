Laborator 4. Programare concurenta in Java

Laborator 4

Programare concurenta in Java

Java a fost primul limbaj de programare important care a inclus suport “direct” pentru fire de executie. Limbajul ofera facilitati pentru crearea firelor de executie, pentru controlul acestora si pentru sincronizare (regiuni critice, notificari). Incepand cu versiunea 1.5, suportul pentru programarea concurenta a fost extins considerabil, printr-o noua biblioteca in care sunt implementate elemente de baza pentru sincronizare (semafoare, bariere, zavoare etc .) si colectii la care accesul concurent este mai eficient (liste, multimi, cozi etc.).

Pentru crearea de noi fire de executie in Java exista doua posibilitati: extinderea clasei Thread sau implementarea interfetei Runnable (aceasta solutie este adoptata de obicei daca extinderea unei clase nu este posibila).

Un model pentru crearea si pornirea unui fir de executie prin extinderea clasei Thread este:

class MyThread extends Thread {

public void run() {

…

sleep(100);

…

}

}

…

MyThread mt = new MyThread();

…

Un model pentru crearea si pornirea unui fir de executie prin implementarea interfetei Runnable este:

class MyModule extends Module implements Runnable {

public void run() {

…

Thread t = Thread.currentThread();

t.sleep(100);

…

}

}

…

MyModule mm = new MyModule();

Thread mmt = new Thread(mm);

…

Pentru a invata notiunile de baza ale programarii concurente in Java va recomandam urmatoarele surse de documentatie:

  • http://book.javanb.com/java-concurrency-in-Practice/toc.html - Java concurrecy in practice

  • “Introduction to Java threads”, de Brian Goetz
  • “Concurrency in JDK 5.0”, de Brian Goetz

Probleme clasice de sincronizare

Producator – Consumator

Aceasta problema se refera la doua procese (sau fire de executie), dintre care unul (producatorul) insereaza date intr-un buffer iar celalalt (consumatorul) extrage date din buffer. Buffer-ul are o dimensiune limitata, deci pentru functionarea corecta a acestui model trebuie asigurate urmatoarele conditii:

  • producatorul sa nu poata insera date daca buffer-ul este plin

  • consumatorul sa nu poata extrage date daca buffer-ul este gol

  • producatorul si consumatorul sa nu actioneze simultan asupra buffer-ului

In plus, trebuie sa ne asiguram ca in functionarea acestor doua procese nu vor aparea situatii de deadlock (aceste situatii provin din erori de programare avand ca efect blocarea atat a consumatorului, cat si a producatorului, fara posibilitatea de a se debloca unul pe celalalt).

Cititori – Scriitori

Problema cititori – scriitori descrie situatia in care mai multe fire de executie sau procese au acces la o zona de memorie comuna. Unele fire de executie doresc sa scrie date in zona de memorie, iar alte fire de executie doresc sa citeasca date din zona de memorie. Este permis ca mai multe fire de executie care citesc din zona de memorie sa aiba acces simultan la aceasta, dar un fir de executie care scrie in zona de memorie nu o poate accesa simultan cu alte fire de executie care scriu sau citesc. Problema are trei variante:

1. (Readers preference) In aceasta varianta, pentru a optimiza modul de acces la memorie, daca un fir de executie doreste sa citeasca din zona de memorie in timp ce si alte fire de executie fac operatii de citire, i se va da acces imediat la memorie (chiar daca la momentul respectiv exista si fire de executie care asteapta pentru operatia de scriere). In acest fel sunt dezavantajate firele de executie care doresc acces pentru scriere, care pot astepta la infinit daca “cititorii” cer acces la memorie foarte frecvent.

2. (Writers preference) In acest caz, daca un fir de executie cere acces pentru scriere la memorie, i se va acorda accesul cat mai repede posibil (adica, dupa ce firele de executie care acceseaza zona de memorie in momentul curent vor termina). In aceasta varianta sunt dezavantajate firele de executie care fac operatii de citire.

3. Primele doua variante ale problemei pot duce la situatii de “starvation” (situatia in care un fir de executie asteapta la infinit). De aceea s-a propus o a treia varianta in care se introduce conditia ca timpul de asteptare pentru orice fir de executie sa fie finit. Solutia la aceasta varianta a problemei este mai putin performanta decat in cazul primelor doua variante, dar garanteaza o functionare “corecta” pentru toate firele de executie.

Problema filozofilor

Aceasta problema, care ilustreaza accesul la resurse partajate in sisteme de calcul, descrie 5 filozofi care stau la o masa circulara. Pe masa sunt 5 farfurii (cate una in fata fiecarui filozof) si 5 tacamuri (fiecare filozof are un tacam in stanga si unul in dreapta). In timp ce stau la masa filozofii pot face doua actiuni: mananca sau se gandesc. Pentru a manca, un filozof are nevoie de doua tacamuri – pe care le poate folosi numai daca nu sunt deja luate de catre vecini. Problema cere construirea unui algoritm dupa care sa actioneze filozofii astfel incat sa nu se ajunga la deadlock (situatia in care fiecare filozof tine cate un tacam in mana si asteapta ca vecinul sa elibereze tacamul celalalt).
